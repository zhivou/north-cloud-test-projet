# Important assumptions
- Swag Labs is an early-stage web application.
- `data-test` is a protected test ID used across the development team and treated as a critical element for this test framework. Developers should not remove, move, or modify these elements without notifying the QA team. Changes are allowed only when the E2E framework is updated and validated with the new selectors.
- Swag Labs appears to make no product-related API calls in the tested flows. Most behavior is handled on the client side, and there is no clearly exposed backend API.
- The product list shows only 6 products and has no pagination.
- User sessions expire quickly (around 5 minutes).
- The web application has no way to crate test data.

# Framework structure
The framework has 3 projects: `setup`, `auth-tests` (depends on `setup`), and `general-tests` (depends on `setup`).

## Setup Project
The main purpose is to set preconditions. Depending on how Swag Labs evolves, it can grow in different ways, but the baseline includes test data setup, user setup, system-under-test checks, system cleanup, and framework precondition checks (tests that validate the framework itself). In its current state, the goal is to ensure that the app is available for testing and that a standard user session is captured for other projects to use.

## Auth Tests Project (needs setup)
This project covers all authentication-related tests and does not use pre-created session files, because its purpose is to validate authentication itself. It can run at the same time as setup, but this may create extra noise when the application is completely down.

## General Tests Project (needs setup)
This project has a general scope because it tests the main application functionality. It can be extended or split later based on company direction. Given its current size, all tests not related to authentication are kept in this project. It reuses the `standard_user` session created in setup to avoid logging in repeatedly.

## Add new projects when needed
The framework is ready to be extended with new projects in `playwright.config.ts`.

## Framework layout

```text
├── .auth/
├── .github/
│   └── workflows/
├── components/
├── data-models/
├── fixtures/
├── pages/
├── playwright-report/
├── setup/
├── test-results/
├── tests/
│   ├── auth/
│   └── general/
├── utils/
```

- `.auth/`: Local Playwright storage-state files (saved authenticated sessions for reuse between projects).
- `.github/workflows/`: CI pipeline definitions for running tests.
- `components/`: Reusable UI interaction blocks used by pages.
- `data-models/`: Typed domain models used across tests and pages.
- `fixtures/`: Shared Playwright fixtures and test setup helpers.
- `pages/`: Page Object Model classes for UI interactions.
- `playwright-report/`: HTML reports generated by Playwright runs.
- `setup/`: Setup projects that prepare auth/session preconditions.
- `test-results/`: Raw artifacts produced during test execution.
- `tests/`: Test projects (`auth` and `general`).
- `utils/`: Helpers

## CI and Parralelization
All tests are ready to run both locally and in CI, with the current setup using the maximum available local workers and 3 workers in CI. The CI workflow is intended to be triggered manually as an on-demand regression run and can be found here: https://github.com/zhivou/north-cloud-test-projet/actions/workflows/playwright.yml

## Secrets and .env
For simplicity, this project does not use a `.env` file or any secrets manager.

## External libraries
Only one external library is used: `@faker-js/faker`. It is a useful addition for test data generation and will become even more valuable as the application and test models mature over time.

## Reporting

Reporting is configured in `playwright.config.ts` with `reporter: 'html'`.

- **Local HTML report**: Each run generates `playwright-report/`.
- **Run artifacts**: `test-results/` stores execution artifacts, including traces and videos for failed tests (`trace: 'retain-on-failure'`, `video: 'retain-on-failure'`).
- **CI artifact upload**: The GitHub workflow in `.github/workflows/playwright.yml` uploads `playwright-report/` as an artifact named `playwright-report` (retention: 30 days), even when tests fail (unless the workflow is cancelled).
- **How to open report locally**:
  ```bash
  npx playwright show-report
  ```

## Custom test fixtures for authenticated state

Authenticated state is prepared through the setup project and stored in `.auth/` for reuse.

- **`.auth/`**: Local storage-state directory (one JSON file per user session).
- **`setup/auth.setup.ts`**: Runs setup tests that create authenticated sessions. The active setup test creates a session for `standard_user`; additional user setups are included as skipped examples.
- **`utils/auth.utils.ts`**: Contains `createStorageStateUI()`, which clears previous auth files, logs in through the UI, and saves a fresh Playwright `storageState` JSON into `.auth/`.

## Framework comments and examples
Some methods include comments and examples explaining why a decision was made, to help reviewers better understand the thought process. A few tests are marked with `.skip` for demonstration purposes only.

## Clean Setup/Teardown
Following best practices, this framework uses proactive cleanup in setup and `beforeEach` hooks instead of relying on teardown logic, due to the early state of the application under test. At this stage, dedicated teardown is not necessary, so pre-test cleanup is used consistently across test cases.

# Considerations and Improvemnts
- Many current test cases can be moved to lower test levels: cart calculation, sorting logic, and mocked checkout failures are better suited for unit, component (for example Storybook/Jest), or backend-level tests. Keeping all of them in E2E increases runtime and maintenance cost, so E2E should focus on realistic user workflows (for example: Login -> Sort -> Add products to cart -> Review cart -> Checkout) and prioritize the most important user-facing flows.
- The Swag Labs development team should improve application testability in partnership with QA. Key improvements include adding a reliable way to seed environments, enabling local app startup (with Docker or natively), adopting ephemeral builds in CI/CD, and shifting testing left so automation stays effective, focused, and valuable instead of becoming automation for its own sake.
- Keep the E2E suite lightweight and end-user-centered.
- The test framework can introduce API helpers, for example, login can be done through an API call instead of the UI to reduce session-creation time.
